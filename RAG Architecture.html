01 · Full System Architecture
02 · Dual-Context Retrieval Pipeline
Key design: every RAG query uses two retrieval paths in parallel — Postgres direct inject (latest 8 articles, always fresh) and Qdrant vector search (semantic top-5). Direct context is prepended to the prompt, guaranteeing the LLM sees the newest data even if Qdrant hasn't indexed it yet.
03 · RAG API Endpoints
POST /rag/query

Primary endpoint. Builds dual context (Postgres + Qdrant), streams the LLM answer as Server-Sent Events — one chunk per data: frame, terminated by [DONE].

Always responds — never returns "no information". Falls back to training knowledge when no articles are found, with a note that no recent news was retrieved.

SSE streaming LlamaIndex top-k=5 chunk=512/64
POST /rag/agent_query

Agentic mode. Emits structured __STEP__ progress events before tokens: retrieve → reflect → requery → synthesize. Frontend can render each step as a visible reasoning trace.

Uses agentic_rag.py — a multi-step loop that can re-retrieve if the first pass context is insufficient.

agentic multi-step __STEP__ events
POST /rag/query_once

Non-streaming fallback. Collects all token chunks and returns a single JSON response — for clients or proxies that cannot consume SSE. Same dual-context retrieval as /rag/query.

Returns {"answer": "...full text..."} after the LLM completes generation.

JSON response no-SSE fallback
04 · Design Patterns
Dual-Context RAG (Postgres + Qdrant)

The primary innovation over vanilla RAG: Postgres-direct injection guarantees the LLM always sees the latest articles for a ticker (up to 8), even before Qdrant has indexed them. Qdrant provides semantic breadth across all stored articles. Both contexts are merged before prompt construction.

This eliminates the freshness lag inherent in index-only RAG — new articles are query-available immediately after ingestion, not after the next embedding cycle.

LlamaIndex Qdrant top-5 Postgres latest-8 fused context
Event-Driven Ingestion Pipeline

APScheduler triggers a 5-minute ingestion cycle. New article IDs are pushed to a Redis queue, decoupling ingestion from processing. The processor drains the queue independently, allowing backpressure management.

FinBERT runs in batch inference mode — articles are grouped and scored in a single forward pass, reducing GPU idle time significantly.

APScheduler Redis Queue FinBERT batch
Cache-Aside Pattern (Redis)

/stocks/overview is cached in Redis with a 60-second TTL. On miss, FastAPI queries PostgreSQL, aggregates sentiment per ticker, and writes back to cache. Cache is busted immediately when a watchlist add/remove occurs.

This eliminates N+1 queries to PostgreSQL on every frontend SWR poll, keeping backend latency under 50ms at dashboard load.

Redis TTL 60s cache-aside bust on mutation
Streaming SSE + Graceful Fallback

The RAG analyst streams tokens via SSE, giving sub-second perceived latency. Each token is a plain-text chunk — data: <text>\n\n — not JSON-wrapped. The stream terminates with data: [DONE]\n\n.

If the streaming generator emits no tokens (proxy/provider edge case), the engine falls back to response.response on the LlamaIndex object. /rag/query_once provides a full non-SSE fallback path.

05 · Scaling Considerations & Possibilities
Inference Scaling
▸
vLLM as Ollama drop-in — continuous batching, PagedAttention; 5–10× throughput for concurrent RAG queries
▸
Quantized models (GGUF Q4_K_M) reduce VRAM from 16 GB to 6 GB, enabling RTX 3060 deployment
▸
Swap to Groq API for 500 tok/s vs ~30 tok/s local; zero cold-start latency
▸
FinBERT model cache — avoid reloading weights on each cycle; pin to GPU memory between ingestion runs
Data & Storage Scaling
▸
Qdrant distributed mode — shard collections across nodes; handles 100M+ vectors horizontally
▸
Hybrid search: combine Qdrant vector similarity with BM25 keyword ranking for better precision on ticker symbols
▸
Neon serverless PostgreSQL — autoscaling, branching for dev; no idle cost outside market hours
▸
Redis Cluster + Pub/Sub — high-availability cache; broadcast live sentiment scores to all connected clients
Architecture Possibilities
▸
WebSocket price feed — Finnhub/Alpaca free-tier WS replaces yfinance polling; push live bid/ask to frontend during market hours
▸
Market-hours awareness — NYSE open label (9:30–16:00 ET), disable stale-data warnings, throttle ingestion outside hours
▸
Alert engine — threshold alerts via email/Telegram when bull/bear score crosses ±0.5
▸
Agentic RAG v2 — LangGraph multi-agent: one agent retrieves, one critiques, one synthesizes; tool-calling for live price lookup
